# Анализ современных CRM → самое необходимое для Sales OS

Дата: 2026-07-04. Автор: Claude (по документации и материалам amoCRM/Kommo, Bitrix24, Pipedrive + практика внедрений). Цель: взять механики, которые реально двигают продажи у малого B2B, и отсечь остальное.

---

## 1. Что делает современные CRM рабочими (по документации)

### 1.1 Ответственный — на КАЖДОЙ сущности (amoCRM/Kommo)
У сделки, контакта и компании всегда есть «Ответственный». Вокруг этого построено всё: автораспределение новых заявок по правилам (график работы, вес менеджера), **повторная заявка от существующего клиента → автоматически тому же менеджеру**, при смене ответственного у сделки — каскадная смена у связанных контакта/компании/открытых задач, история «почему заявка попала этому менеджеру».
→ **Подтверждает решение Founder**: ownerUserId обязателен, и не только у Company.

### 1.2 «Нет сделки без задачи» / activity-based selling (Pipedrive)
Ядро Pipedrive: менеджер управляет не исходом (закрытие сделки), а контролируемыми действиями (звонок, письмо, встреча). У каждой активной сделки ВСЕГДА запланировано следующее действие; экран менеджера — не список сделок, а «мои задачи на сегодня». Это главный анти-Excel механизм: Excel не напоминает.
→ Для нас: Task с дедлайном + экран «Сегодня» важнее многих полей. Правило продукта: сделка без следующего шага подсвечивается как «зависшая».

### 1.3 Карточка = вся история, действия из карточки (Bitrix24)
Каждое обращение (звонок, письмо, мессенджер) автоматически падает в карточку клиента; из карточки можно позвонить/написать. Менеджер не ищет историю по блокнотам.
→ Наш Timeline (уже в дизайне CRM-домена) + «быстрые действия» из карточки.

### 1.4 Контроль дублей по телефону/email/ИНН (Bitrix24)
Автоматическое обнаружение и слияние дублей в единую карточку. Для B2B РФ ключ — ИНН (у нас уже unique per org), для контактов — телефон/email.
→ Дешёвая версия: предупреждение «похожий контакт уже есть» при создании.

### 1.5 Messenger-first inbox (Kommo)
Заявки из WhatsApp/Telegram/etc. падают в одну воронку; сделка = карточка с полной перепиской, задачами, напоминаниями. Kommo вырос именно на этом.
→ Подтверждает наш блок Communications (Telegram+MAX) как стратегический — но после ядра.

### 1.6 Почему 70% внедрений проваливаются (практика)
Саботаж менеджеров: CRM усложняет работу вместо упрощения; обязательные поля, которые лень заполнять; ощущение «системы контроля», а не помощника.
→ **Наши принципы:** минимум обязательных полей (имя — и хватит), автозаполнение всего, что можно; каждая функция должна ЭКОНОМИТЬ время менеджеру, а не только давать контроль руководителю. Request-to-Cash (расценка+КП за минуты) — идеальный «пряник».

## 2. Выжимка: что необходимо Sales OS (приоритеты)

| П | Механика | Статус у нас |
|---|----------|--------------|
| **P1** | Ответственный менеджер: Company + Contact (`ownerUserId`, member org), default = создатель; повторный клиент → тот же менеджер (позже, при inbox) | **← следующая задача** |
| **P1** | Tasks: тип (звонок/написать/встреча), дедлайн, привязка к contact/company; экран «Сегодня» | нет (модуль tasks пуст) |
| **P2** | Deal pipeline: kanban, этапы, сумма; правило «нет сделки без задачи» (подсветка зависших) | нет (после фидбека) |
| **P2** | Дубли: предупреждение по телефону/email при создании контакта; ИНН уже unique | нет |
| **P3** | Timeline карточки (заметки+звонки+задачи в одну ленту) | частично (notes) |
| **P3** | Быстрые действия из карточки (позвонить/написать) | нет |
| стратег. | Messenger inbox Telegram+MAX; автораспределение заявок | backlog (DECISIONS) |

**Не делаем** (по ANTI_GOALS, подтверждено анализом): проекты, склад, HR, вики, корпчат, сложные обязательные поля, тяжёлые отчёты на старте.

## 3. Следующий шаг

Фича «ответственный менеджер» (решение Founder 2026-07-03) — спецификация в NEXT.md: `ownerUserId` на Company И Contact сразу (одинаковый паттерн, один PR), валидация членства в организации (как resolveCompanyId), default = текущий пользователь, фильтр «мои» в списках. После неё — Tasks (P1 №2).
